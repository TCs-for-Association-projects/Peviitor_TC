- **`config/epics-and-stories.json`** is the single source of truth for epics, user stories, testing types, sections, environments, and statuses.
- **Authoring** a test case вҶ’ open an Issue with the **Test Case** form. Auto-label syncs labels from the form fields.
- **Executing** a test case вҶ’ post a **comment** with slash commands. Never edit the issue body after creation.
- **Reporting** вҶ’ the Test Matrix workflow regenerates the QA dashboard and publishes it to GitHub Pages.

---

## Authoring a test case

1. **Issues вҶ’ New issue вҶ’ Test Case**.
2. Fill in the form (Epic, User Story, Section, Environment, steps, etc.).
3. Submit. Auto-label syncs `epic:`, `story:`, `type:`, `section:`, `env:` labels and adds `needs-review` if the Epic and User Story don't match.

**Title format:**
```
TC - [Feature] - [Action] - [Expected Result]
```
_Example:_ `TC - Footer links - Hover and click - Correct pages open`

---

## Executing a test case

Execution happens **in comments**, not by editing the issue body. The Test Execution workflow parses slash commands and syncs labels.

| Command | What it does |
|---|---|
| `/status passed` | вң… Mark as passed |
| `/status failed #123` | вқҢ Mark as failed and link bug `#123` |
| `/status blocked` | рҹҹЎ Mark as blocked |
| `/status partially-passed` | рҹҹ  Partial pass |
| `/status not-run` | вҡӘ Reset to not run |
| `/bug #123` | рҹҗӣ Add `bug-found` label + link bug issue |
| `/note <text>` | рҹ“қ Log a note without changing status |
| `/cross-os` | Toggle the `cross-os` flag |
| `/cross-browser` | Toggle the `cross-browser` flag |

Multiple commands can appear in one comment. The bot replies with a formatted execution summary including the status transition, bug link, and a dashboard link.

> вҡ  When marking a test as `failed`, **always include a bug reference** (`#123` or a URL) so the failure can be traced.

## Dashboard

Regenerated by the **Test Matrix** workflow on every issue event plus nightly at 03:00 UTC.

**Pages:**

| Route | What you'll find |
|---|---|
| **`#/overview`** | KPI cards, status breakdown bar, epic health chart, distribution donuts (environment, type, section, OS, browser, user story). |
| **`#/tests`** | Full filterable/sortable table. Filters: search, epic, story, status, section, env, type, author, **assignee**. Click any row to expand its description and metadata. Filter state persists in the URL. |
| **`#/coverage`** | Interactive requirements traceability matrix and gap analysis, plus a Section Г— Testing Type heatmap and assignee workload overview. |
| **`#/guide`** | Built-in beginner-friendly documentation with JSON copy-paste blocks for first-time configuration and maintenance. |

**Exports:** the footer offers `в¬Ү CSV` (Excel-ready, UTF-8 BOM, 26 columns including status icon, assignees, days open, days since update, description excerpt) and `в¬Ү JSON` (raw data feed).

**Keyboard:** press `/` anywhere to focus the global search. `Esc` closes it.

**Theme:** dark by default, switchable via the moon/sun toggle in the top bar. Choice persists in `localStorage`.

## Label taxonomy

All labels are created idempotently by `scripts/bootstrap-labels.js` (run **Bootstrap Labels** workflow manually after cloning). Colors are standardized.

| Prefix / name | Meaning | Managed by |
|---|---|---|
| `Test_Case` | Gating label вҖ” required for all workflow processing | Issue template |
| `epic: F1`, `epic: S1`, вҖҰ | Epic assignment | Auto-label |
| `story: US1`, вҖҰ | User Story assignment | Auto-label |
| `type: Navigation`, вҖҰ | Testing type | Auto-label |
| `section: Footer`, вҖҰ | Website section under test | Auto-label |
| `env: Staging`, вҖҰ | Target environment | Auto-label |
| `status: Passed` / `Failed` / `Blocked` / `Not run` / `Partially passed` | Execution outcome | **Test Execution** (slash commands) |
| `bug-found` | A bug was linked during execution | Test Execution |
| `cross-os`, `cross-browser` | Test should cover multiple OSes / browsers | Form + Test Execution toggles |
| `needs-review` | Auto-label flagged a mismatch (e.g. Epic/US) | Auto-label |

**Epics (11):** F1вҖ“F4 (Footer), S1вҖ“S6 (Search), E1 (generic).
**User stories (25):** see `config/epics-and-stories.json` for the full list.

To add or rename any of the above, **edit `config/epics-and-stories.json`** and push вҖ” the Generate Template workflow regenerates `.github/ISSUE_TEMPLATE/test_case.yml` automatically, and the next Bootstrap Labels run will sync label colors/descriptions.

## Workflows

| File | Trigger | Purpose |
|---|---|---|
| `Auto_Label_Test_Case.yml` | Issue opened/edited/labeled | Sync form-derived labels, flag mismatches |
| `Test_Execution.yml` | Issue comment created | Parse slash commands, update status/bug/cross-* labels, post summary |
| `Test_Matrix.yml` | Issue events, nightly 03:00 UTC, manual | Regenerate `docs/index.html` + JSON + CSV, commit to `main` |
| `Generate_Template.yml` | Push to `config/epics-and-stories.json` or the generator | Regenerate `.github/ISSUE_TEMPLATE/test_case.yml` |
| `Bootstrap_Labels.yml` | Manual or Push to config | Create/update all managed labels from config (idempotent) |

All commit-producing workflows `git pull --rebase origin main` before pushing to avoid race conditions when several fire back-to-back.

## First-time setup for a new repository

1. Push this codebase.
2. **Actions вҶ’ Bootstrap Labels вҶ’ Run workflow.** Creates ~50 managed labels with correct colors.
3. **Settings вҶ’ Pages вҶ’ Deploy from `main` / `docs`.** Enables the QA dashboard.
4. **Actions вҶ’ Test Matrix вҶ’ Run workflow.** Generates the initial dashboard.
5. Open an issue using the Test Case form to verify the full pipeline end-to-end.
