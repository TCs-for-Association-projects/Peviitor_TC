# Peviitor_TC

**QA test management for [peviitor.ro](https://peviitor.ro/)** — a job search platform. This repository contains no application source code; every test case is a **GitHub Issue**, and every workflow here exists to turn those issues into a living, filterable dashboard.

## How it works

```
┌───────────────────┐   authoring     ┌──────────────────┐
│  Issue form       │────────────────▶│  GitHub Issue    │
│  (test_case.yml)  │                 │  labelled Test_* │
└───────────────────┘                 └──────────────────┘
         ▲                                     │
         │ regenerates from                    │ auto-label
         │                                     ▼
┌───────────────────┐                 ┌──────────────────┐
│ config/epics-and- │                 │  Labels synced   │
│ stories.json      │                 │  (epic/story/…)  │
└───────────────────┘                 └──────────────────┘
                                              │
                                   execution  │ slash commands
                                              ▼
                                      ┌──────────────────┐
                                      │  Status, bug,    │
                                      │  cross-platform  │
                                      │  labels updated  │
                                      └──────────────────┘
                                              │
                                   matrix job │
                                              ▼
                                      ┌──────────────────┐
                                      │  docs/index.html │
                                      │  Dashboard SPA   │
                                      └──────────────────┘
```

- **`config/epics-and-stories.json`** is the single source of truth for epics, user stories, testing types, sections, environments, and statuses.
- **Authoring** a test case → open an Issue with the **Test Case** form. Auto-label syncs labels from the form fields.
- **Executing** a test case → post a **comment** with slash commands. Never edit the issue body after creation.
- **Reporting** → the Test Matrix workflow regenerates the QA dashboard and publishes it to GitHub Pages.

## Authoring a test case

1. **Issues → New issue → Test Case**.
2. Fill in the form (Epic, User Story, Section, Environment, steps, etc.).
3. Submit. Auto-label syncs `epic:`, `story:`, `type:`, `section:`, `env:` labels and adds `needs-review` if the Epic and User Story don't match.

**Title format:**
```
TC - [Feature] - [Action] - [Expected Result]
```
_Example:_ `TC - Footer links - Hover and click - Correct pages open`

## Executing a test case

Execution happens **in comments**, not by editing the issue body. The Test Execution workflow parses slash commands and syncs labels.

| Command | What it does |
|---|---|
| `/status passed` | ✅ Mark as passed |
| `/status failed #123` | ❌ Mark as failed and link bug `#123` |
| `/status blocked` | 🟡 Mark as blocked |
| `/status partially-passed` | 🟠 Partial pass |
| `/status not-run` | ⚪ Reset to not run |
| `/bug #123` | 🐛 Add `bug-found` label + link bug issue |
| `/note <text>` | 📝 Log a note without changing status |
| `/cross-os` | Toggle the `cross-os` flag |
| `/cross-browser` | Toggle the `cross-browser` flag |

Multiple commands can appear in one comment. The bot replies with a formatted execution summary including the status transition, bug link, and a dashboard link.

> ⚠ When marking a test as `failed`, **always include a bug reference** (`#123` or a URL) so the failure can be traced.

## Dashboard

Regenerated by the **Test Matrix** workflow on every issue event plus nightly at 03:00 UTC.

**Pages:**

| Route | What you'll find |
|---|---|
| **`#/overview`** | KPI cards, status breakdown bar, epic health chart, distribution donuts (environment, type, section, OS, browser, user story). |
| **`#/tests`** | Full filterable/sortable table. Filters: search, epic, story, status, section, env, type, author, **assignee**. Click any row to expand its description and metadata. Filter state persists in the URL. |
| **`#/coverage`** | Interactive requirements traceability matrix and gap analysis, plus a Section × Testing Type heatmap and assignee workload overview. |
| **`#/guide`** | Built-in beginner-friendly documentation with JSON copy-paste blocks for first-time configuration and maintenance. |

**Exports:** the footer offers `⬇ CSV` (Excel-ready, UTF-8 BOM, 26 columns including status icon, assignees, days open, days since update, description excerpt) and `⬇ JSON` (raw data feed).

**Keyboard:** press `/` anywhere to focus the global search. `Esc` closes it.

**Theme:** dark by default, switchable via the moon/sun toggle in the top bar. Choice persists in `localStorage`.

## Label taxonomy

All labels are created idempotently by `scripts/bootstrap-labels.js` (run **Bootstrap Labels** workflow manually after cloning). Colors are standardized.

| Prefix / name | Meaning | Managed by |
|---|---|---|
| `Test_Case` | Gating label — required for all workflow processing | Issue template |
| `epic: F1`, `epic: S1`, … | Epic assignment | Auto-label |
| `story: US1`, … | User Story assignment | Auto-label |
| `type: Navigation`, … | Testing type | Auto-label |
| `section: Footer`, … | Website section under test | Auto-label |
| `env: Staging`, … | Target environment | Auto-label |
| `status: Passed` / `Failed` / `Blocked` / `Not run` / `Partially passed` | Execution outcome | **Test Execution** (slash commands) |
| `bug-found` | A bug was linked during execution | Test Execution |
| `cross-os`, `cross-browser` | Test should cover multiple OSes / browsers | Form + Test Execution toggles |
| `needs-review` | Auto-label flagged a mismatch (e.g. Epic/US) | Auto-label |

**Epics (11):** F1–F4 (Footer), S1–S6 (Search), E1 (generic).
**User stories (25):** see `config/epics-and-stories.json` for the full list.

To add or rename any of the above, **edit `config/epics-and-stories.json`** and push — the Generate Template workflow regenerates `.github/ISSUE_TEMPLATE/test_case.yml` automatically, and the next Bootstrap Labels run will sync label colors/descriptions.

## Workflows

| File | Trigger | Purpose |
|---|---|---|
| `Auto_Label_Test_Case.yml` | Issue opened/edited/labeled | Sync form-derived labels, flag mismatches |
| `Test_Execution.yml` | Issue comment created | Parse slash commands, update status/bug/cross-* labels, post summary |
| `Test_Matrix.yml` | Issue events, nightly 03:00 UTC, manual | Regenerate `docs/index.html` + JSON + CSV, commit to `main` |
| `Generate_Template.yml` | Push to `config/epics-and-stories.json` or the generator | Regenerate `.github/ISSUE_TEMPLATE/test_case.yml` |
| `Bootstrap_Labels.yml` | Manual or Push to config | Create/update all managed labels from config (idempotent) |

All commit-producing workflows `git pull --rebase origin main` before pushing to avoid race conditions when several fire back-to-back.

## Scripts (local use)

Requires **Node.js ≥ 20**. No dependencies to install — all scripts use built-in APIs only.

```bash
# Regenerate the issue form from config
npm run generate-template

# Regenerate the dashboard (requires GITHUB_TOKEN with issues:read)
GITHUB_TOKEN=ghp_…  GITHUB_REPOSITORY=TheodorIvascu/Peviitor_TC  npm run generate-matrix

# Offline preview with mock data
MOCK_ISSUES=path/to/mock.json  GITHUB_TOKEN=x  GITHUB_REPOSITORY=x/y  npm run generate-matrix
```

`npm run auto-label` and `npm run bootstrap-labels` exist for debugging but are normally invoked by workflows.

## First-time setup for a new repository

1. Push this codebase.
2. **Actions → Bootstrap Labels → Run workflow.** Creates ~50 managed labels with correct colors.
3. **Settings → Pages → Deploy from `main` / `docs`.** Enables the QA dashboard.
4. **Actions → Test Matrix → Run workflow.** Generates the initial dashboard.
5. Open an issue using the Test Case form to verify the full pipeline end-to-end.
